在Typescript里extends Error的终极方案

在 Typescript/Javascript（以下用 TS/JS 简称）的项目中，如何正确的处理异常是一个很常见的问题。一般认为的最佳实践是，创建一个项目独有的根 Error class，再在其上继承出子 Error class，用以区分原生/框架/业务逻辑异常。

  • 直接 extends

class FooError extends Error {}
console.log(new FooError() instanceof FooError); // false，匹配错误
1
2

原因

在 TS Github Repo 中的这个wiki。

对于 Error,Array 等原生 class，他们的 constructor 使用了new.target来调整原型链。但在 es5 中无法保证new.target一定存在，所以会导致继承出来的 class 在原型链上缺失。 对于new.target，在 mdn 上相关它的文章介绍。

TS 官方的建议是使用如下的代码：

class FooError extends Error {
  constructor(m: string) {
    super(m);

    // Set the prototype explicitly.
    Object.setPrototypeOf(this, FooError.prototype);
  }
}

这段代码手动的将FooError加回了实例的原型链中。但也存在一个很大的问题，就是所有的子类都必须再用这种方式手动设置原型链。该方法无疑不是最终方案。 而且，这样处理还有两个小问题：

class FooError extends Error {
  constructor(m: string) {
    super(m);
    Object.setPrototypeOf(this, FooError.prototype);
  }
}
class BarError extends FooError {
  constructor(m: string) {
    super(m);
    Object.setPrototypeOf(this, BarError.prototype);
  }
}
console.log(new BarError('I am BarError').stack);
// Error: I am BarError
//     at BarError.FooError [as constructor] (******)
//     at new BarError (******)
//     at Object.<anonymous> (******)
console.log(new Error('I am Error').stack);
// Error: I am Error
//    at Object.<anonymous> (******)

注意到：

相较于原生 Error，BarError和FooError的constructor被加入到了 call stack 中，这样不理想，多层继承会给 debug 带来很多噪音。 尽管我们的 class 叫做BarError，但是 stack 的开头，异常的名字还是Error。这给我们在 debug 中寻找问题的源头造成了困难。

改善

修复原型链

上面提到，官方给出的修复原型链方案并不完美，存在着需要层层修复的问题。那么有没有什么办法一劳永逸呢？ 这里就要回到之前提到过的new.target。它指向的是 new 的这个 class 最外层的 constructor。既然如此，就可以利用它来修复原型链

class FooError extends Error {
  constructor() {
    super('');
    Object.setPrototypeOf(this, new.target.prototype);
  }
}
console.log(new FooError() instanceof FooError); // true
console.log(new FooError() instanceof Error); // true

class BarError extends FooError {}

console.log(new BarError() instanceof BarError); // true
console.log(new BarError() instanceof FooError); // true
console.log(new BarError() instanceof Error); // true

可以看到，尽管只在最内层修复了原型链，整个原型链也是完整的。这显然比官方推荐的方法优雅很多。 由于并不是所有环境都支持Object.setPrototypeOf，在这些环境中，可以用__proto__来替代

class FooError extends Error {
  constructor() {
    super('');
    if (typeof Object.setPrototypeOf === 'function') {
      Object.setPrototypeOf(this, new.target.prototype);
    } else {
      (this as any).__proto__ = new.target.prototype;
    }
  }
}

修复 stack

这里可以使用 V8 中独有的Error.captureStackTrace来重新生成 call stack。

if (typeof (Error as any).captureStackTrace === 'function') {
  (Error as any).captureStackTrace(this, new.target);
}

同样，这里使用了new.target来避免继承带来的层层修复问题。 虽然这个方法只在 V8 环境中有用，但对于 NodeJs 项目作用的很大。

修复 name

这里的解决方案依旧是new.target。

this.name = new.target.name;

总结

将以上三种办法结合在一起的最终解决方案：

// 使用 BaseError 当作 base Error class
class BaseError extends Error {
  constructor(message: string) {
    super(message);
    this.name = new.target.name;
    if (typeof (Error as any).captureStackTrace === 'function') {
      (Error as any).captureStackTrace(this, new.target);
    }
    if (typeof Object.setPrototypeOf === 'function') {
      Object.setPrototypeOf(this, new.target.prototype);
    } else {
      (this as any).__proto__ = new.target.prototype;
    }
  }
}

class BarError extends BaseError {}
console.log(new BarError('') instanceof BarError); // true
console.log(new BarError('I am BarError').stack);
// BarError: I am BarError
//     at Object.<anonymous> (******)
