命名原则

命名原则

平台的所有命名，在无特殊情况下，优先依照业界最佳实践或约定俗成的规范进行，但基于平台的功能设计，仍然有部分命名方式存在特例。 存在特殊考虑的场景，会遵循到如下几个因素：

特例

  • 后端兼容，如有必要的情况下尽量与底层数据库字段保持一致，避免中间转换或映射
  • 面向用户群体的英文水平不一，尽量减轻词汇的记忆负担
  • 取词倾向于自然用语
  • 同义或近义词汇，在不造成歧义的情况下，以短优先
  • 服务端实例文件夹以 xxxSvr 格式为准，中间禁止任何符号，且必须以 Svr 结尾（大小写敏感）

示例

场景选用词汇非选用词汇取舍原则
父级IDpidparentId以短优先，统一命名
子元素subschildren以短优先，统一用复数
中文名称namezhchineseName以短优先，统一命名
备注字段memonote/remarks以短优先，统一命名
键值命名id/codekey/title后端兼容
键值命名（title为中文的场景）id/namezhkey/title后端兼容
键值命名（key为字符的场景）code/namezhkey/title后端兼容

约定

区分环境类型

工程内，无论前中后端程序，编译时均通过 process.env.NODE_ENV 来区分当前处于开发模式还是生产模式 若 process.env.NODE_ENV 不等于 dev ，则判定为生产环境，否则判定为开发环境，

环境常量

开发或生产环境，分别用以下常量字符进行区分（小写）：
开发环境：dev
生产环境：prod
--------------------------------------
判定方式：
process.env.NODE_ENV  !==  'dev'  ?  '生产环境'  :  '开发环境'
--------------------------------------
环境生成：
在开发环境，（命令行 或 CLI）触发调试指令，如 yarn dev 时，会注入：
process.env.NODE_ENV  ===  dev
由此构造出开发环境，其他情形，

  • 也许有注入
    process.env.NODE_ENV  ===  prod
  • 也许未注入任何值，导致：
    process.env.NODE_ENV  ===  undefind
    均被视为生产环境

环境判断函数

globals.isProd();
// 返回 true 表示生产环境
// 返回 false 表示开发环境

URL 路径书写

URL路径无论是配置还是参数进行书写时，均不对 ( 起始、或结尾 ) 之处是否添加了反斜杠进行范式要求，而由调用函数自行预处理。若因此导致的运行时异常，视为 BUG 级的 ISSUE，而非配置级，或传参级错误。

约束

ESModule

tCoffe 全局使用了 ESModule 模式，因此在 package.json 文件中，请一直确保 type = module

export/import VS module.export/require

考虑到对 ES6+ 以及 tree shaking(摇树)的支持，项目中全程使用 export/import 的包引入模式，而不要使用 module.export/require

代码风格

Js/Ts 全局代码风格约束

  1. 语句强制分号结尾
  2. 对象内属性末项禁止尾逗号
  3. 除非 if (foo) return something 的单行风格的写法，否则， if/else不能省略花括弧
  4. 禁止代码内 console.log，如需要，则自定创建对应的 // eslint-disable no-console，或对具体行应用 eslint-disable-line
  5. 代码文件末行以空行结束

建议

参考

  • tCoffe 采用了 yarn workspace 仓库管理模式，因此建议将 yarn 作为主要包管理工具。

磁盘/文件 路径书写范式（适用于：无论是配置，还是参数）

对于文件路径，无论针对 Windows 系统，还是 MaxOX/Linux，均建议统一采用反斜杠  /  例如： X/:foo/bar， 作为路径片段分隔符，对于配置文件，若坚持对 windows 路径使用正斜杠  \  例如： X:\foo\bar，则需使用转义符  \  例如： X:\\foo\\bar
